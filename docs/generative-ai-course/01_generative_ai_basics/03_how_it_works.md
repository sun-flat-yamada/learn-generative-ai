# 1.3 生成AIの仕組み

> **本セクションの目標:** LLMの基本的な動作原理を理解し、なぜ特定の振る舞いをするか説明できる。

---

## 本セクションの狙い

生成AIが「魔法の箱」ではなく、膨大なデータの統計的学習と高度なアーキテクチャによって構築された「数学的モデル」であることを理解します。Transformerの基礎から、2026年現在の主流である推論モデルやエージェントの仕組みまでを学び、AIの能力と限界を正しく判断できる基礎知識を養います。

### 期待される学習効果

1. Transformerの核となる「Attention機構」の概念を理解する。
2. 推論モデル（Reasoning Models）がどのように論理的思考を実現しているか説明できる。
3. エージェント（Agentic AI）やネイティブ・マルチモーダルといった最新トレンドの技術的背景を把握する。

---

すべての主要LLMは「Transformer」アーキテクチャをベースにしています。2025年以降は、計算効率を高める **Mixture of Experts (MoE)** や、長文処理に強い **State Space Models (SSM/Mamba)** といった派生技術との組み合わせが主流になっています。

```txt
入力テキスト
    ↓
[トークナイザー] — 文字列 → トークン（数値）に分割
    ↓
[Embedding層] — トークンを高次元ベクトル空間にマッピング
    ↓
[Self-Attention機構] ← Transformerの核心（文脈の理解）
    ↓
[Feed Forward層 / MoE層] — 必要な「専門家回路」だけを動かして高速化
    ↓
[出力層] — 次のトークンの確率分布を計算
    ↓
出力テキスト（最も確率が高いトークンを選択し続ける）
```

> 📚 参考: [Vaswani et al., "Attention Is All You Need" (2017)](https://arxiv.org/abs/1706.03762)

---

## Self-Attention（自己注意機構）とは

```txt
文章: 「太郎は東京で働いている。彼は毎日電車で通勤する。」

「彼」= 誰を指すのか？
→ Attention は「太郎」との関連性が高いことを学習
→ 文脈を理解した翻訳・要約・回答が可能に
```

Self-Attention によって、LLM は文章全体の文脈を把握できます。

## LLMのトレーニングプロセス

### Phase 1: 事前学習（Pre-training）

```txt
インターネット上の膨大なテキスト（兆単位のトークン）
    ↓
「次のトークンを予測する」タスクで学習
    ↓
世界の知識を圧縮したパラメータが生成される
```

### Phase 2: ファインチューニング（Fine-tuning / RLHF）

```txt
人間によるフィードバック（良い回答/悪い回答の評価）
    ↓
RLHF（人間のフィードバックからの強化学習）
    ↓
指示に従いやすく、有害な出力を減らす
```

> 📚 参考: [OpenAI, "Reinforcement Learning from Human Feedback"](https://openai.com/research/learning-to-summarize-with-human-feedback)

### Phase 3: 推論の最適化（Reasoning Optimization） [2025-2026 Trend]

```txt
複雑な問題に対する「思考の連鎖（Thinking Process）」
    ↓
強化学習によって「正解にたどり着く思考パターン」を自己学習
    ↓
OpenAI o1 / DeepSeek-R1 等の「推論モデル」の誕生
```

## 🧠 推論モデル（Reasoning Models）の仕組み

2025年以降、LLMは「即答する」から「考えてから答える」へと進化しました。

1. **Thinking Tokens（思考トークン）**: 回答を出す前に、内部的に試行錯誤を行い、論理的なステップを組み立てます。
2. **自己修正**: 思考の途中で間違いに気づくと、自分でルートを修正します。
3. **計算資源の動的配分**: 簡単な質問には即答し、難しい数学やプログラミングの問題には時間をかけて思考します。

> 💡 **ポイント**: これにより、これまで不得意だった複雑な論理パズルや高度な数式、バグの多いコードの修正能力が飛躍的に向上しました。

## 🤖 Agentic AI：行動するAIの仕組み

最新のAIは「書く」だけでなく「行う」能力を持っています（エージェント化）。

- **思考・実行ループ (Reasoning-Action Loop)**:
  「計画」→「ツール選択（検索、ブラウザ、コード実行）」→「実行」→「結果の評価」を自律的に繰り返します。
- **自律的な目標達成**:
  「明日の出張の準備をして」という曖昧な指示に対し、カレンダーの確認、飛行機の予約、現地の天気予報のチェック、必要な資料の作成などを一気通貫で実行します。

---

## 🎨 ネイティブ・マルチモーダルの仕組み

最新モデル（Gemini 2.0/3.0, GPT-5等）は、テキスト・画像・音声を一つのモデルで統合的に処理する**ネイティブ・マルチモーダル**へと進化しました。

- **共通の感覚**: テキストも画像も音声も、同じ種類の「データ（トークン）」として一度に処理します。これにより、以前のような「言葉を画像に変換する別のAI」が不要になりました。
- **クロスモーダルな理解**: 「動画のこのシーンで鳴っている音を楽譜にして」といった、視覚・聴覚・言語を またぐ 高度な指示に直接対応可能です。

## 💥 TIPS — AIモデルの提供コストとトレンド

AIへの質問は「無料」に見えますが、裏側では膨大な計算コストが発生しています。

- **1日あたりの運用コスト**: 約 **70万ドル（約1億円）**（2023年時点の推計）
- **1クエリ（質問）あたりのコスト**: 約 **0.36セント（約0.5円）**

> 📖 **出典:**
>
> - [Reddit / SemiAnalysis via Dylan Patel (2023)](https://www.reddit.com/r/artificial/comments/12whu0c/chatgpt_costs_openai_700000_a_day_to_keep_it/)

あなたが「今日のランチは何がいい？」と聞くだけで、OpenAIはサーバー代として約0.5円を支払っている計算になります。

> 最も無駄にコストがかかるのは、最後に「ありがとう」と付け加えることです。
> これを行うと、それまでの全てのやり取りをもう一度LLMに読み込ませて、それに対してありがとうの意味を解釈する処理が走るので、使わない結果にそのセッションで最大の処理入力がかかることになります。

2026年現在は効率化が進んでいますが、高度な推論モデルには依然としてコストがかかります。

- **推論効率の大幅な向上**: 標準的なモデル（GPT-4o/Gemini 1.5レベル）のAPI価格は、2023年比で **1/10以下**（100万トークン数円〜数十円）まで低下しました。
- **推論モデル（Reasoning）の別料金**: OpenAI o1やDeepSeek-R1といった「考えるAI」は、内部で生成される「思考トークン」に対しても課金されます。
- **「ありがとう」のコストと環境負荷**: セッションが長くなるほど、過去の全てのやり取りが記憶（コンテキスト）として再計算されるため、不必要なやり取りは計算資源と電力の浪費に繋がります。

> 📖 **参考:**
>
> - [DeepSeek-V3/R1 Pricing](https://www.deepseek.com/)
> - [OpenAI API Pricing](https://openai.com/api/pricing)

## 💡 TIPS: 「確率的に正しい」問題

> LLM は **次のトークンの確率を計算して出力を生成します。**
> つまり「最も可能性が高い」回答を生成するのであって、「事実として正しい」回答を生成するわけではありません。
>
> これがハルシネーション（幻覚）の根本原因です。
> 「もっともらしい嘘」は「もっともらしい事実」と同じメカニズムで生成されます。
>
> **AIを使うときは必ず事実確認を！**

## テキスト生成の仕組み（サンプリング）

LLMは一度にすべての文章を生成するのではなく、**1トークンずつ生成**します。

```python
# 概念的なコード
context = "日本の首都は"

# 各トークンの確率を計算
probabilities = {
    "東京": 0.95,
    "大阪": 0.03,
    "京都": 0.01,
    ...
}

# Temperature で確率分布を調整
# Temperature = 0.0: 常に最高確率のトークンを選ぶ（決定的）
# Temperature = 1.0: 確率分布に従ってサンプリング（ランダム性あり）
# Temperature = 2.0: より多様・創造的だが不安定

next_token = sample(probabilities, temperature=0.7)
# → "東京" を選択 → コンテキストに追加 → 繰り返す
```

## RAG（Retrieval-Augmented Generation）の仕組み

LLMは学習時点以降の知識を持ちません。RAGはこの問題を解決します。

```txt
ユーザーの質問
    ↓
[ベクトル検索] — 社内DB/ドキュメントから関連情報を検索
    ↓
[コンテキスト注入] — 検索結果をプロンプトに追加
    ↓
[LLM] — 注入された情報を元に回答生成
    ↓
最新・正確な情報を含む回答
```

> 📚 参考: [AWS: What is RAG?](https://aws.amazon.com/what-is/retrieval-augmented-generation/)

## パラメータ数とモデル能力の関係

モデルの「賢さ」はパラメータ数、学習データ量、そして「推論時の計算量」に相関します。

```txt
パラメータ ≈ AIの「記憶容量」「思考回路の複雑さ」

GPT-3:   175B params   → 高品質文章生成
GPT-4:   推定1T+ params → 多段階推論、専門知識
Llama 3: 8B〜70B params → ローカル実行可能
Llama 3/4 (8B):    軽量、スマホやPCでも動作可能
GPT-4 / Gemini 1.5: 兆単位 (1T+) 推定。高度なマルチモーダル対応
DeepSeek-V3 (671B): MoEアーキテクチャ。巨大だが、動くのは一部（37B）で超高速
```

モデル名の後ろに **120B** などと記載されているものは、パラメータ数を示しています。

```txt
gpt-oss-120b
gpt-oss-20b
  :
```
