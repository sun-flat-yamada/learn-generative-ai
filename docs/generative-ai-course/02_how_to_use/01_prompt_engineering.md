# 2.1 プロンプトエンジニアリング

> **本セクションの目標:** 効果的なプロンプトの書き方をマスターし、AIから最大限の成果を引き出す。

---

## 本セクションの狙い

**「英語」**は世界の共通語ですが、これからのビジネスの最強の共通語は**「プロンプト（AIへの指示）」**です。本セクションでは、AIの能力を100%引き出すための対話技術（プロンプトエンジニアリング）と、目的に応じた適切なツールの選び方を学びます。基本から、2026年現在の主流である推論モデルの最適化、エージェントワークフローの設計手法までを体系的に学習し、「AIに指示を出す」段階から、「AIと共に業務プロセスを設計する」段階へのステップアップを目指します。

### 期待される学習効果

1. **CRAFTフレームワーク**を用いた、構造的で精度の高いプロンプト作成スキルの習得。
2. **推論モデル（Reasoning Models）**の特性を理解し、その能力を最大限に引き出す手法の習得。
3. **XMLタグ**や**エージェント的アプローチ**を活用した、より高度で安全な業務自動化設計の理解。

## Markdown を使ったプロンプトの書き方

### なぜ Markdown が有効か？

LLMは大量の Markdown 形式のテキスト（GitHub、Qiita、Stack Overflow 等）で学習しているため、
Markdown の構造を理解して処理できます。

> 📚 参考: [YC: Best prompts use Markdown](https://www.reddit.com/r/LLMDevs/comments/1ljdul6/yc_says_the_best_prompts_use_markdown/)

---

### Markdown 記法とプロンプトでの使いどころ

#### `#` `##` `###` — 見出し（ヘッダー）

**役割:** プロンプトを論理的なセクションに分ける

```markdown
# タスク
以下のメールを日本語に翻訳してください。

# 入力テキスト
[英語メールの内容]

# 出力要件
- 丁寧語を使用
- 敬称は「様」
- 200字以内
```

**ポイント:** `#` が多いほど深い階層（H1 > H2 > H3）。重要な見出しは `#` で区切る。

---

#### ` ``` ` — コードブロック

**役割:** コード・データ・フォーマットを他のテキストと明確に分離する

```markdown
以下のPythonコードのバグを修正してください。

\```python
def calculate_average(numbers):
    total = sum(numbers)
    return total / len(numbers)  # ← ゼロ除算の可能性
\```

期待する動作: 空のリストの場合は 0 を返す
```

**ポイント:** 言語名（`python`, `json`, `sql` 等）を指定すると構文ハイライトのヒントになる。

---

#### `-` `*` `1.` — リスト

**役割:** 複数の条件・要件を明確に列挙する

```markdown
以下の条件でメールを作成してください:
- 受信者: 上司（田中部長）
- 目的: 来週の会議の延期依頼
- 締め切り: 今日中に送信
- トーン: 丁寧かつ簡潔

NG事項:
1. 個人的な理由の詳細は含めない
2. 代替日程は3つ提案する
3. 本文は300字以内
```

---

#### `**太字**` `*イタリック*` — 強調

**役割:** 重要なキーワード・制約を強調する

```markdown
顧客向けの製品説明文を作成してください。

**対象読者:** 技術知識のない60代の方
**製品:** スマートフォンの使い方
*トーン: やさしく、専門用語なし*
```

---

#### `>` — ブロック引用

**役割:** 入力データ・文脈情報を本文と区別する

```markdown
以下の顧客レビューを3点に要約してください。

> この製品は使いやすく、バッテリーも長持ちです。
> ただ、重さが少し気になりました。
> デザインは他の製品より好みです。

要約形式: 箇条書き3点
```

## 効果的なプロンプトの構成要素（CRAFT フレームワーク）

```txt
C - Context（文脈）: AIに背景情報を伝える
R - Role（役割）:    AIにどの専門家として振る舞うか指示
A - Action（行動）:  具体的に何をしてほしいか
F - Format（形式）:  出力フォーマットを指定
T - Target（対象）:  誰向けの内容か
```

### 実例: CRAFT を使ったプロンプト

```markdown
# Context（文脈）
私は新入社員向けの研修資料を作成中です。
対象者はIT初心者の大学卒業者です。

# Role（役割） ... 最近は役割よりも条件を明示することでより精度が高められると言われています
あなたは10年以上の経験を持つITエンジニアです。
専門用語を噛み砕いて説明することが得意です。

# Action（行動）
「クラウドコンピューティング」を説明してください。

# Format（形式）
- 見出しで構造化する
- 具体的な日常の例えを使う
- 図の代わりにASCIIアートで概念図を示す
- 500字以内

# Target（対象）
IT知識がほぼない新入社員（22〜23歳）
```

## Chain-of-Thought（思考の連鎖）プロンプト

複雑な問題を解かせるときは、**考えるプロセスを明示**させると精度が上がります。

```markdown
【普通のプロンプト】
「1000円の商品を20%割引、さらに消費税10%を追加した価格は？」

【Chain-of-Thought プロンプト】
「1000円の商品を20%割引、さらに消費税10%を追加した価格は？
ステップごとに計算過程を示してください。」
```

→ 計算ミスや論理エラーが大幅に減少します

> 📚 参考: [Google: Chain-of-Thought Prompting Elicits Reasoning in LLMs](https://arxiv.org/abs/2201.11903)

## 2026年の新常識：推論モデル（Reasoning Models）へのプロンプティング

ChatGPT (o1/o3), Claude 3.7, Gemini 2.5 Pro などの「推論モデル」は、内部で思考プロセス（思考の連鎖）を自動的に行います。

### 推論モデルでのコツ

1. **指示はシンプルに:** 「ステップバイステップで考えて」と強く指示する必要はありません。モデルが自動で行います。
2. **目的と制約を明確に:** 「どう考えるか（プロセス）」よりも「何を達成したいか（ゴール）」と「守るべきルール（制約）」を詳細に伝えます。
3. **思考を邪魔しない:** 過度なプロンプトのテクニックを詰め込みすぎると、モデル本来の推論能力を制限してしまう場合があります。

## XMLタグによる構造化

プロンプトを `<context>`, `<task>`, `<constraints>` などの XML タグで区切る手法は、2026年現在、多くのモデル（特に Claude シリーズ）で非常に効果的です。

### メリット

- **可読性の向上:** 人間にとってもAIにとっても、どこが指示でどこがデータか明確になります。
- **入力インジェクションの防止:** ユーザー入力を XML タグ（例: `<user_input>`）で囲むことで、ユーザーの入力が命令として誤認されるのを防げます。

```markdown
# 高度なプロンプト例
<role>
あなたはプロのテクニカルライターです。
</role>

<task>
以下の <source_text> を、<target_audience> に合わせて要約してください。
</task>

<target_audience>
ITに詳しくない20代の若手社員
</target_audience>

<source_text>
[ここに長い文章を貼り付け]
</source_text>
```

## Agentic Reasoning（エージェント的プロンプティング）

単一の回答を得るのではなく、AIに「ツールを使い、計画を立て、実行する」プロセスを指示します。

- **思考、行動、観察（ReAct）:** AIに「まず計画、次に実行、その結果を自己評価」させるループを作ります。
- **ワークフローの定義:** 「Aを確認し、もしBならCを実行せよ」といった条件分岐を含む指示を与えます。

---

## Few-shot プロンプト（例示プロンプト）

AIに「こういう感じで答えてほしい」という例を示します。

```markdown
以下のパターンで、商品の口コミを評価してください。

例1:
入力: 「軽くて持ち歩きやすい。でも電池が切れるのが早い」
出力: 長所: 携帯性 ★★★★★ / 短所: バッテリー ★★☆☆☆

例2:
入力: 「デザインがおしゃれで気に入っているが少し高め」
出力: 長所: デザイン ★★★★★ / 短所: コスパ ★★☆☆☆

では次を評価してください:
入力: 「操作が直感的でわかりやすい。耐久性が少し不安」
```

## よくある失敗パターンと改善例

| 失敗パターン         | 問題                   | 改善例                                                     |
| -------------------- | ---------------------- | ---------------------------------------------------------- |
| 「要約して」         | 何を、どのくらい短く？ | 「100字以内で3点に要約して」                               |
| 「良い文章に直して」 | 何が良い文章？         | 「ビジネスメールの書式に直して。丁寧語使用、200字以内」    |
| 「コードを書いて」   | 言語は？ 仕様は？      | 「Python 3.12でCSVを読み込む関数を書いて。エラー処理含む」 |
| 長い情報を全部貼る   | トークン無駄遣い       | 必要な情報だけを構造化して提供                             |

## 💡 TIPS: プロンプトエンジニアリングの「神プロンプト」構造

> 最高品質の出力を得るための黄金律:
>
> 1. **役割を与える**（「あなたは専門家として...」） (注釈: この方式は近年、効果が薄いという報告もあります。条件を明確に与える方が効果的です。)
> 2. **文脈を説明する**（「現在の状況は...」）
> 3. **具体的な指示を出す**（「〜を行ってください」）
> 4. **出力フォーマットを指定する**（「箇条書きで...」）
> 5. **制約を明示する**（「〜は含めないこと」）
>
> これだけで回答品質が劇的に向上します。
> また、Claude は XML タグ（`<task>`, `<context>`）を、OpenAI は Markdown が特に効果的という報告があります。
>
> 2026年現在は、プロンプトを「プログラム」として最適化する **DSPy** のようなフレームワークも普及しており、手動でプロンプトを微調整する時代から、データに基づいて自動最適化する時代へと移行しつつあります。
>
> 📚 参考: [Prompt Engineering Guide](https://www.promptingguide.ai/)
