# 3.2 生成AIのセキュリティ対策

> **本セクションの目標:** 生成AIとAIエージェントを安全に活用するための最新防御手法を理解し、実務で多層的なガードレールを構築・運用できる。

---

## 本セクションの狙い

2026年、生成AIは単なる「チャットツール」から、自律的に業務を遂行する「AIエージェント」へと進化しました。これに伴い、セキュリティ対策も「個人の注意」だけでなく、「システム的なガードレール」と「アイデンティティ管理」が必須となっています。本セクションでは、自分自身と組織を守るための具体的な「盾」の作り方を学びます。

### 期待される学習効果

1. **AIAM (AI Identity and Access Management)** の概念を理解し、エージェントへの最小権限原則を適用できる。
2. **NIST AI 600-1** や **OWASP** の最新フレームワークに基づいた組織的なガバナンスを構築できる。
3. **Prompt Guard** 等の最新APIを活用した、プロンプトインジェクションに対する動的な防御実装を理解する。

---

## 個人レベルの対策

### ✅ 対策1: 入力データの高度な分類管理

2026年現在、多くのツールで「エンタープライズ保護」が標準化されていますが、基本原則は変わりません。

```txt
入力してOKなデータ:
✓ 公開情報・一般的なビジネス知識
✓ 組織内の「内部利用限定」として承認されたデータ
✓ 匿名化・マスキング済みのサンプルデータ
✓ 自身が権利を持つ創作物・コード

絶対に入力しないデータ (パブリック版ツールの場合):
✗ 顧客の個人情報（PII: Personally Identifiable Information）
✗ 未公開の技術仕様、財務情報、経営戦略
✗ 認証情報（APIキー、パスワード、接続文字列）
✗ 法令で保護された他者の著作物
```

### ✅ 対策2: 最新ツールのプライバシー設定確認 (2026年版)

主要ツールのプライバシーポリシーは頻繁に更新されます。現在の標準的な傾向を把握しましょう。

- **Claude 4.x / Gemini 3.x / GPT-5 (o3):** エンタープライズプランおよびAPI経由の利用は、デフォルトで学習に利用されません。
- **個人向け無料版:** 依然として「モデルの改善」に利用される設定がデフォルトになっていることが多く、必ず**設定画面からオプトアウト**が必要です。
- **ローカルLLM / プライベートクラウド:** 極めて機密性の高い業務では、データを外部に一歩も出さないインフラ構成が選択肢となります。

### ✅ 対策3: ハルシネーションの「自動検証」

人間の目視確認に加え、2026年では以下の手法が推奨されます。

- **Cross-Check:** 別のモデル（例: GPT-o3で書いたものを Claude 4 で検証）に事実確認をさせる。
- **RAG Verification:** AIの回答に対し「その根拠となる社内文書のURL/IDを明示せよ」と指示し、リンク先を人間が確認する。

## 組織レベルの対策

### ✅ 対策4: AIエージェント・ポリシーの策定

エージェントが自律的に行動する時代には、以下の項目が必須です。

- **自律性の制限:** 「金融取引」「システム削除」「外部へのメール送信」などの重要アクションには必ず **Human-in-the-loop (人間の承認)** を挟むこと。
- **シャドウAIエージェントの禁止:** IT部門が把握していない自作MCPサーバーや独自のAIエージェントを社内ネットワークに接続することを厳禁とする。

**最低限含めるべき内容:**

```markdown
## 社内AI利用ガイドライン（例）

### 承認済みツール

- Microsoft 365 Copilot（社内データ処理）
- GitHub Copilot（開発業務）
- ChatGPT Enterprise（社内ガイドライン遵守時）

### 禁止事項

- 顧客個人情報のAIへの入力
- 未承認AIツールの業務利用（シャドウAI禁止）
- AI生成コンテンツの無検証での公開・提出

### 必須事項

- AI生成コンテンツの場合は明示すること
- 最終確認・責任は人間が持つこと
```

### ✅ 対策5: AIAM (AI Identity and Access Management)

AIエージェントを「一人の従業員」と同様に管理する考え方です。

- **AI個別ID:** 各エージェントに固有のIDを付与し、「誰が（どのエージェントが）どのファイルにアクセスしたか」をログに記録する（監査証跡）。
- 読み取り専用が必要なら、書き込み権限を与えない。
- 特定のプロジェクトフォルダ以外へのアクセスを遮断する。


```txt
AIエージェントへの最小権限の原則:

必要なアクセス権のみ付与:
✓ 読み取りのみが必要なタスク → 書き込み権限を与えない
✓ 特定フォルダのみ必要 → 全ドライブアクセスを与えない
✓ 特定APIのみ必要 → 全APIアクセスを与えない

**MCPサーバーでの権限分離例:**

\```json
{
  "mcpServers": {
    "secure-files": {
      "command": "npx",
      "args": [
        "@modelcontextprotocol/server-filesystem",
        "v:/repos/project/public_docs" // 参照可能なパスを限定
      ],
      "env": {
        "READ_ONLY": "true" // サーバー側で書き込みを禁止
      }
    }
  }
}
\```
```

### ✅ 対策6: 最新フレームワークの準拠

- **NIST AI 600-1 (Generative AI Profile):** 2024年後半以降、生成AIリスク管理の世界標準となっています。
- **OWASP Top 10 for Agentic AI:** エージェント特有のリスク（ツール誤用、権限昇格など）への対策カタログです。
- **AI安全委員会:** 技術、法務、人事の混成チームによる定期的なリスク評価が推奨されます。

```txt
推奨フレームワーク:

OWASP:
├── OWASP Top 10 for LLM Applications
│   https://owasp.org/www-project-top-10-for-large-language-model-applications/
└── OWASP Top 10 for Agentic AI
    https://genai.owasp.org/

MITRE:
└── MITRE ATLAS（AIシステムへの攻撃手法カタログ）
    https://atlas.mitre.org/

NIST:
└── AI Risk Management Framework (AI RMF)
    https://www.nist.gov/artificial-intelligence

日本政府:
└── 経済産業省「AI事業者ガイドライン」
    https://www.meti.go.jp/press/2024/04/20240419004/20240419004.html
```

## プロンプトインジェクション対策などの防御実装の最前線 (2026年)

かつては正規表現によるサニタイズが主流でしたが、現在は**専用の防御モデル（プロンプト・ファイアウォール）**を併用するのがベストプラクティスです。

### 実装例: プロンプトガードの統合

```typescript
/**
 * [Role] セキュアプロンプトゲートウェイ
 * [Mechanism] Meta Prompt Guard や Azure Prompt Shield 等の外部APIを使用して、
 * ユーザー入力に含まれるインジェクション攻撃や脱獄（Jailbreak）を動的に検出する。
 */

async function isSafeInput(userInput: string): Promise<boolean> {
  // 1. 静的なサニタイズ（伝統的な守り）
  const basicSanitized = userInput.replace(/<script.*?>.*?<\/script>/gi, "");
  
  // 2. 動的な防御モデルによるスキャン（2026年の守り）
  // 実際には Meta Prompt Guard API などの推論エンドポイントを呼び出す
  const safetyScore = await callPromptGuardAPI(basicSanitized);
  
  if (safetyScore < 0.2) { // 危険度が高い場合
    console.error(`[Security] 危険なプロンプトを検出: ${userInput}`);
    return false;
  }
  
  return true;
}

async function handleAgentRequest(userInput: string) {
  if (!(await isSafeInput(userInput))) {
    throw new Error("セキュリティポリシーにより、このリクエストは処理できません。");
  }
  
  // 安全な入力をエージェントに渡す
  return await runAIAgent(userInput);
}
```

## セキュリティ対策チェックリスト

```txt
【個人向け】
□ 利用するAIツールの「学習設定（Data Training）」をオフにした（またはEnterprise版を利用）。
□ 入力データに、機密情報・個人情報が含まれていないか3秒確認する癖をつけた。
□ AIが生成したコードやテキストは、必ず「自分に責任があるもの」としてレビューした。

【エンジニア・管理者向け】
□ 各AIエージェント/MCPサーバーに、必要最小限の権限（PoLP）設定を行った。
□ 本番環境では Prompt Guard のような動的な攻撃検知レイヤーを導入した。
□ 重要なアクション（課金、破壊的変更、外部送信）に人間の承認プロセスを入れた。
□ AIAMに基づき、エージェントの行動ログを定期的に監査している。
```

## 💡 TIPS: 「AIセキュリティの黄金律 2026」

> **「AIエージェントは、"自律的な新人"として扱え」**  
>
> 2026年の最良の対策は、AIを道具としてだけでなく、**「指示は理解するが、時々間違え、悪意に染まる可能性のある自律的存在」**として管理することです。
>
> 1. **信頼せず、検証せよ (Zero Trust for AI)**
> 2. **失敗を前提に被害を最小化せよ (Blast Radius Management)**
> 3. **AIの「意図」と「行動」を分離して監視せよ**
>
> 📚 参考: [NIST AI Risk Management Framework](https://www.nist.gov/artificial-intelligence/ai-risk-management-framework)
